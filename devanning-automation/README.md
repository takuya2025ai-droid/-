# デバンニング事業 業務自動化（Google Apps Script）

請求書・支払明細・シフト表がバラバラに手入力されている状態を解消するための、
Googleスプレッドシート＋Apps Scriptの自動化一式です。

以前お渡ししたExcelプロトタイプ(`devanning_automation_prototype.xlsx`)と同じデータ構造を、
実運用できる形（Googleスプレッドシート＋自動連携）に発展させたものです。

## できること

1. **カレンダー表示の自動生成** — ③案件実績・④シフト実績を入力すると、今まで見ていたカレンダー形式のシフト表を自動で作り直す
2. **取引先フォーム → ③案件実績への自動追記** — 毎週の依頼メールをGoogleフォームに置き換え、回答が自動で反映される
3. **請求書・支払明細のExcel自動出力** — 毎月1日に先月分を自動集計し、Excel(.xlsx)としてDrive保存＋メール送付
4. **LINE連携** — 出発・到着・業務終了の報告をボタンで受け取り記録、毎日15時の翌日確認、毎週金曜のシフト配信、毎週水曜の提出リマインドを自動化

## ファイル構成

```
apps-script/
  appsscript.json     マニフェスト(タイムゾーン・Webアプリ設定)
  Config.gs           シート名・列位置などの共通定義
  Bootstrap.gs         初回セットアップ用（シート雛形とサンプルデータを一括作成）
  Pricing.gs           ①②単価マスタを検索して請求額・支払額を計算する関数(CALC_BILLING/CALC_PAYOUT)
  AutoFill.gs           ③④に手入力した瞬間に自動計算の数式をセットする(onEdit)
  Calendar.gs          カレンダー表示の自動生成
  FormIntake.gs        取引先フォーム回答 → ③案件実績への自動追記
  Summary.gs           ⑤請求書サマリ・⑥支払明細サマリの月次集計
  ExcelExport.gs       ⑤⑥をExcel(.xlsx)化してDrive保存・メール送付
  Line.gs              LINE Messaging API連携(Webhook・リマインド配信)
  Triggers.gs          カスタムメニュー・時間主導トリガーの一括登録
```

## まずはここだけ確認してください（非エンジニア向けの動作確認手順）

コードが正しく動くかどうかは、以下の5分でできる手順だけで確認できます。フォームやLINEはまだ関係ありません。

1. [sheets.google.com](https://sheets.google.com) で新しい空のスプレッドシートを開く
2. メニューの「拡張機能」→「Apps Script」をクリック
3. `apps-script/` フォルダの中の `.gs` ファイルを1つずつ開き、中身を全部コピーする →
   Apps Scriptエディタで「ファイル」の＋ボタンから同じ名前のスクリプトファイルを作り、貼り付ける
   （最初からある `コード.gs` は使わないので中身を空にするか削除してよい）
   ※ `appsscript.json` は、まず歯車アイコン「プロジェクトの設定」を開き「"appsscript.json" マニフェスト ファイルをエディタで表示する」にチェックを入れると、左側のファイル一覧に出てくるので、そこに貼り付ける
4. 上部の関数選択プルダウンで `setupSpreadsheet` を選び、▶(実行)ボタンを押す
5. 「承認が必要です」という画面が出たら、「権限を確認」→自分のGoogleアカウントを選択→
   「詳細」→「(プロジェクト名)に移動」→「許可」の順にクリックする
   （これは自分で作ったスクリプトを自分のシートで動かすための確認で、危険な操作ではありません）
6. 実行が終わると「初期セットアップが完了しました」という案内が出る
7. 元のスプレッドシートのタブに戻って画面を更新する → ①〜⑧のシートが8枚できているはずです
8. `③案件実績_請求元` を開く → サンプルの1行が入っていて、H列(20F本数)・I列(40F本数)・J列(請求額)に
   自動で数字(2、1、95000)が入っていれば成功
9. `④シフト実績_支払元` を開く → サンプルの3行が入っていて、F列(担当本数=1)・G列(支払額=4000)が
   自動で入っていれば成功
10. 試しに③に新しい行を1行手入力してみる（取引先名は「サンプル取引先A」、現場は「サンプル現場」のままでOK。
    日付・合計フィート・本数だけ変えてみる）→ Enterキーを押した瞬間にH〜J列が自動で埋まれば、
    仕組みが正しく動いています

ここまで確認できたら、サンプル行は削除し、①②に実際の単価表を入力してから、③④に本物の実績を入力し始めてください。
フォーム連携・Excel自動出力・LINE連携は、この土台が問題なく動くことを確認してから次に進めるのがおすすめです。

## セットアップ手順（本番運用に向けて）

### 1. スプレッドシートとスクリプトの用意

1. 新しいGoogleスプレッドシートを作成する
2. 「拡張機能」→「Apps Script」を開く
3. `apps-script/` 内の各 `.gs` ファイルの中身を、同名のスクリプトファイルとしてコピー＆ペーストする
   （`appsscript.json` は、Apps Scriptエディタの「プロジェクトの設定」→「"appsscript.json" マニフェスト ファイルをエディタで表示する」を有効にすると編集できます）
4. Apps Scriptエディタの関数選択で `setupSpreadsheet` を選び、実行する
   → ①〜⑧の必要なシートと見出しが一括作成されます
5. スプレッドシートを開き直すと「デバンニング自動化」という独自メニューが追加されています

### 2. 単価マスタの入力

- `①単価マスタ_請求` に、共有いただいた単価表(全取引先分)を転記する。取引先名・現場名の組み合わせごとに行を追加していく
  - 本数ベースの取引先(ダイセー型)：20F/40Fそれぞれ1〜4本目以降の単価を入力
  - 人工ベースの取引先(Netforce型)：人工数に応じた単価を1〜4本目以降の列に入力（実質「人工数」の意味で使う）
  - 同じ取引先でもエリア/現場によって単価が違う場合(画像の柏・藤代 vs 三郷など)は、現場名を変えて別の行として追加する
  - 行数の上限はないので、25社分すべてこの1枚に追加していけばそのまま使えます(`CALC_BILLING`が取引先名＋現場名で検索します)
- `②単価マスタ_支払` に、個人事業主・協力会社それぞれの本数目別の支払単価を入力する

### 3. 日々の運用に③④シートを使う

- `③案件実績_請求元`：取引先からの依頼（現場・合計フィート・本数・人工）を1日1レーン1行で入力（後述のフォーム経由で自動化可能）
- `④シフト実績_支払元`：その日その現場に入った作業者を1人1行で入力
- どちらも入力するとH〜J列・G列に自動で20F/40F本数・請求額・支払額が計算されます
- 「デバンニング自動化」メニュー →「カレンダーを今すぐ更新」で、見慣れたカレンダー表示を再生成できます

### 4. 取引先フォームの用意

1. 新しいGoogleフォームを作成し、以下の順番・型で質問を追加する

   | 質問 | 型 |
   |---|---|
   | 取引先名 | プルダウンまたは記述式 |
   | 現場名 | 記述式 |
   | レーン | 記述式（任意） |
   | 対象日 | 日付 |
   | 合計フィート | 記述式(数値) |
   | 本数(合計) | 記述式(数値) |
   | 人工希望人数 | 記述式(数値・任意) |
   | 備考 | 記述式（任意） |

2. フォームの「回答」タブ→スプレッドシートアイコン→このスプレッドシートを回答先に指定
3. Apps Scriptエディタの「トリガー」画面で、実行する関数 `onJobFormSubmit` / イベントの種類「フォーム送信時」を追加
   （または `Triggers.gs` の `installFormTrigger_(フォーム編集URL)` を一度だけ実行してもよい）
4. 主要取引先から順に、このフォームでの依頼に切り替えてもらう。フォーマットに応じてもらえない取引先は、当面③に手入力を続ければ混在運用できる

### 5. スクリプトプロパティの設定（機密情報）

Apps Scriptエディタ →「プロジェクトの設定」→「スクリプト プロパティ」に以下を登録する。

| プロパティ名 | 内容 |
|---|---|
| `NOTIFY_EMAIL` | 請求書・支払明細やエラー通知の送付先メールアドレス |
| `DRIVE_EXPORT_FOLDER_ID` | Excel出力の保存先Google DriveフォルダID |
| `LINE_CHANNEL_ACCESS_TOKEN` | LINE Developersで発行するチャネルアクセストークン(長期) |
| `LINE_CHANNEL_SECRET` | LINE Developersのチャネルシークレット(現状のdoPostでは未使用だが将来のため保存推奨) |

### 6. LINE公式アカウントの用意

1. [LINE Developers](https://developers.line.biz/) で Messaging API チャネルを作成する
2. チャネルアクセストークン(長期)を発行し、上記スクリプトプロパティに保存
3. Apps Scriptエディタ →「デプロイ」→「新しいデプロイ」→種類「ウェブアプリ」
   - 実行するユーザー: 自分
   - アクセスできるユーザー: 全員
   - デプロイ後に発行されるURLを控える(第三者に共有しない)
4. LINE Developersの「Messaging API設定」→Webhook URLに3のURLを登録し、Webhookの利用をON
5. 「応答メッセージ」機能はOFFにする(自動応答はこのスクリプトが行うため)
6. 外注さん全員にLINE公式アカウントを友だち追加してもらい、追加直後に何かひとこと送ってもらう
   → `⑦作業者マスタ` に「【要紐付け】表示名」として自動で行が追加されるので、
     A列(作業者名)を実名に書き換えて紐付けを完了する

### 7. 自動化トリガーの一括登録

「デバンニング自動化」メニュー →「自動化トリガーを一括登録」を1回実行する。

登録される自動実行：

| タイミング | 内容 |
|---|---|
| 毎日 6:00 | カレンダー表示を最新化 |
| 毎日 15:00 | 翌日勤務者へLINEで現場確認＋稼働報告ボタンを送信 |
| 毎週水曜 9:00 | 全作業者へシフト提出(空き状況)のリマインドLINE |
| 毎週金曜 9:00 | 翌週シフトを本人ごとにLINEで個別配信 |
| 毎月1日 9:00 | 先月分の請求書・支払明細を自動集計しExcel出力・メール送付 |

取引先フォームのトリガーだけは手順4で個別に登録してください。

## 既知の制約・注意点

- **LINEのWebhook署名検証について**：Apps ScriptのWebアプリ(`doPost`)はHTTPヘッダーを取得する手段がないため、
  LINEが送る `X-Line-Signature` を使った正規のHMAC検証はできません。デプロイURLを外部に公開しない運用でリスクを抑えています。
  より厳密な検証が必要な場合は、Cloud Functions等ヘッダーを扱えるサーバーレス環境への切り出しを検討してください。
- **本数の按分ルール**：「その日の人数＝本数」の場合は全員同額（1本目単価）を自動適用します。
  1人が複数本を担当した例外時は、`④シフト実績_支払元` の「担当本数」列を手動で書き換えてください。
  人数と本数が食い違う日は、その担当本数の合計が案件実績の本数と一致しているか目視確認することをおすすめします。
- **請求書・支払明細のExcelレイアウト**：現状は「取引先/作業者・期間・合計金額」のシンプルな集計表です。
  御社所定の請求書・支払明細書フォーマット(頭書き・振込先・押印欄など)に転記する一手間は残ります。
  自動出力された金額を確認のうえ、正式フォーマットに反映してください。
- **請求書サマリ・支払明細サマリは月次で確定値として書き込む**設計です(SUMIFSの生数式ではなく、集計した値をその場で書き込む)。
  締めた後に③④を修正しても、発行済みの金額は変わりません。再集計したい場合はメニューから「再集計」を実行してください。
